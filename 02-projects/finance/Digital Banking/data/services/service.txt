disputes.json
üõ†Ô∏è Key Implementation Rules
    -Referential Integrity:
       -transaction_id: Strictly references IDs found in transactions.json that have a status of FAILED.
       -customer_id: Mapped via the account_id of the original transaction to ensure the right user owns the dispute.
    -Reasoning: Distributed across FRAUD, WRONG_AMOUNT, DUPLICATE, and OTHER to simulate real-world customer complaints.
    
    -Lifecycle
      
      -OPEN: Under investigation.
      -RESOLVED: Successfully closed.
      -REJECTED: Request denied (e.g., if the user was mistaken).

     -Time Period: All records were created within the last 6 months to maintain consistency with your transaction history.


üí° Dashboard Use Case
   -Transaction Detail View: When a user clicks on a "Failed" transaction, you can check this JSON to see if an active dispute exists and display its status.
   -Notification System: Trigger an alert for the user when a dispute status changes from OPEN to RESOLVED.
   -Admin Portal: Build a "Resolution Queue" for bank staff to view all OPEN disputes and add resolution_notes.
